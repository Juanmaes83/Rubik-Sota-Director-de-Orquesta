Rubik Sota Director de Orquesta — Strategy
1. Decisión estratégica

Rubik Sota Director de Orquesta no debe convertirse en una colección de productos sueltos.

La dirección correcta no es construir:

un producto para restaurantes 3D;
un producto para hoteles con postales;
un producto para escaparates según clima;
un producto para inmobiliarias;
un producto para marketing automático;
un bot comercial separado.

La dirección correcta es construir un solo sistema madre:

Rubik Sota Director de Orquesta
Un motor capaz de convertir datos reales, imágenes reales y contexto real de un negocio en previews comerciales hiperpersonalizadas, listas para enseñar, enviar, escanear, medir y vender.

Los casos de restaurantes, hoteles, inmobiliarias, moda, retail, reformas o negocios locales no son productos independientes. Son verticales de aplicación del mismo motor.

2. Tesis central

El mercado no necesita más demos genéricas.

El valor diferencial está en enseñar a cada negocio una preview personalizada de lo que podría tener:

su plato convertido en menú 3D;
su hotel convertido en campaña visual;
su escaparate adaptado a lluvia, calor o frío real;
su vivienda presentada como tour inmersivo;
su colección de moda convertida en showroom interactivo;
su local convertido en una experiencia visual vendible.

La clave comercial es esta:

“No te vendo una idea abstracta. Te enseño una preview de tu propio negocio, con tus propias imágenes, tus propios datos y una propuesta visual concreta.”

4. Qué NO es

Rubik Sota Director de Orquesta no es:

una simple web;
una demo aislada;
una galería de experimentos;
un repositorio de prototipos IA;
un generador de imágenes sueltas;
un chatbot comercial;
una herramienta de diseño sin salida de ventas;
una colección de módulos sin conexión.

Tampoco debe fragmentarse en productos separados demasiado pronto.

5. Principio de arquitectura

Todo debe depender de un núcleo común.

Datos reales
→ Assets reales
→ Preview personalizada
→ Landing / demo
→ QR / envío
→ Tracking
→ CRM / acción comercial

Cada vertical cambia la forma visual, pero no cambia el sistema base.

7. Núcleo común del sistema

El sistema debe dividirse en bloques reutilizables.

7.1 Asset Library

Biblioteca de imágenes, vídeos, modelos y recursos reales.

Debe permitir:

subir fotos reales por sector;
clasificar assets;
asociar assets a verticales;
distinguir entre asset real, generado, demo e ilustrativo;
preparar fallback visual cuando no haya imagen real;
evitar depender de placeholders.

Estructura sugerida:

assets/
  fashion/
  real-estate/
  food/
  hotels/
  storefronts/
  communities/
  shared/
7.2 Effects Library

Biblioteca de efectos visuales reutilizables.

Origen posible:

CodePen;
GSAP;
partículas;
transiciones;
hover effects;
page transitions;
reveals;
efectos de escaparate;
efectos de showroom;
efectos de before/after.

Estructura sugerida:

effects/
  raw-codepen/
  cleaned/
  production-ready/
  manifest.json

Cada efecto debe documentar:

nombre;
origen;
dependencias;
uso recomendado;
riesgo de rendimiento;
compatibilidad móvil;
estado;
módulos donde aplica.

Regla:

Ningún efecto debe integrarse en producción sin limpieza, aislamiento y QA.

7.3 Manifest System

Cada preview debe tener un manifest propio.

Ejemplo conceptual:

{
  "id": "business-preview-001",
  "vertical": "hotel",
  "businessName": "Example Hotel",
  "location": "Alicante",
  "sourceAssets": [],
  "previewType": "visual-campaign",
  "qrTarget": "",
  "tracking": {
    "enabled": true
  },
  "status": "draft"
}

El manifest permite que diferentes partes del sistema lean la misma fuente:

visor;
landing;
postal;
QR;
CRM;
propuesta comercial.

10. MVP recomendado

No empezar por automatización total.

MVP recomendado:

Crear una preview manual hiperpersonalizada para un negocio real.

Características:

un negocio;
datos reales;
fotos reales o cargadas manualmente;
landing privada;
QR;
propuesta clara;
seguimiento en CRM.

Vertical recomendada:

inmobiliaria;
local business;
hotel boutique;
comunidad / mantenimiento.

No empezar con 3D food como MVP principal.

Motivo:

más riesgo técnico;
más dependencia de calidad 3D;
más tiempo de ajuste;
menor velocidad de validación comercial.
11. Reglas estratégicas
Regla 1

No crear productos separados antes de validar el motor.

Regla 2

Cada vertical debe usar el mismo núcleo:

Data → Assets → Preview → QR/Landing → Outreach → Tracking → CRM
Regla 3

Nada debe parecer IA, demo o prototipo.

Debe parecer:

personalizado;
real;
terminado;
presentable;
comercial.
Regla 4

Todo asset debe tener origen claro.

Clasificación:

real;
generado;
editado;
ilustrativo;
pendiente de sustitución.
Regla 5

Automatizar solo después de validar manualmente.

Regla 6

Cada módulo debe ayudar a vender.

Si no genera una acción comercial, es solo un experimento visual.

12. Riesgos
Riesgo 1 — Fragmentación

Crear demasiados productos separados puede romper el foco.

Mitigación:

mantener un solo sistema madre;
documentar verticales como aplicaciones;
evitar nombres de producto independientes al principio.
Riesgo 2 — Falta de assets reales

Sin fotos reales, muchas previews parecerán artificiales.

Mitigación:

crear Asset Library;
preparar carga manual;
usar fallbacks visuales de calidad;
sustituir placeholders por assets reales cuanto antes.
Riesgo 3 — Exceso de automatización

Automatizar antes de validar puede quemar tiempo y dinero.

Mitigación:

MVP manual;
validar 5-10 demos;
automatizar solo pasos repetidos.
Riesgo 4 — Legal / derechos de imagen

Usar fotos públicas de terceros puede tener restricciones.

Mitigación:

revisar condiciones;
priorizar fotos propias o autorizadas;
documentar origen;
etiquetar contenido como preview ilustrativa cuando aplique.
Riesgo 5 — Promesas no ejecutables

Una preview puede mostrar mejoras que el proveedor real no puede cumplir.

Mitigación:

diferenciar preview comercial de presupuesto técnico;
no inventar datos;
conectar con proveedores reales cuando sea necesario.
Riesgo 6 — Performance

Efectos, vídeos y 3D pueden degradar móvil.

Mitigación:

QA móvil obligatorio;
no abusar de GSAP;
optimizar GLB;
lazy loading;
fallback estático.

No construir cinco productos.

Construir un motor.

Los verticales vendrán después.
